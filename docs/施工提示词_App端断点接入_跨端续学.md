## 2. 复用的后端表(已建好,只读其结构)

```
reading_progress (
  user_id    uuid  default auth.uid(),
  space_id   uuid  references spaces(id) on delete cascade,
  unit_id    uuid  references units(id)  on delete cascade,
  anchor     text,                          -- 单元内的位置锚点(见 §3)
  updated_at timestamptz default now(),
  primary key (user_id, space_id)           -- 每个用户每个空间一行
)
```
- 主键 `(user_id, space_id)`:每个空间只保留**一条最新**进度,写入用 **upsert(冲突即更新)**,last-write-wins。
- `unit_id` = 读到哪个单元(粗位置);`anchor` = 单元内读到哪(细位置)。
- RLS 已限制 `user_id = auth.uid()`,App 带会话即自动隔离。

---

## 3. anchor 约定(开工第一件核对的事 · 跨端跳准的关键)

`unit_id` 决定跳到哪个单元;**`anchor` 决定落在单元内的哪个位置**。两端必须用同一套 anchor 语义,否则会"跳对单元、落错位置"。

**任务**:先确认**网页端当前写入的 anchor 是什么格式**(查网页仓库,或问我)。可能是:
- (a)`units.content` 数组里的**块序号**(如 `"7"` = 第 7 个块),或
- (b)某个 `h2` 的 DOM id(网页特有)。

**推荐统一为 (a) 块序号**:两端都渲染同一份 `content[]` 数组,"第 N 块"在手机和网页上指向同一段内容,跨平台最稳;h2 的 DOM id 是网页特有的,App 没有对应物。

处理:
- 若网页端已是块序号 → App 直接按块序号读写,完成。
- 若网页端是 h2 id → 二者择一统一;推荐**网页端改成块序号**(改动极小),App 也用块序号。把这点作为结论报告给我确认后再实现。
- App 写入时:记录"当前视口顶部最近的那个块的序号"。读取跳转时:滚动到该序号对应的块。

> 不确定就先报告你查到的网页端格式 + 你的对齐建议,等我拍板,别擅自定。

## 4. 读写逻辑

### 4.1 写入(阅读时)
- 在单元阅读页,**防抖**(停止滚动约 1–2s,或离开页面/切后台时)做一次 `upsert`:
  `{ space_id, unit_id, anchor }`(`user_id` 由默认值 `auth.uid()` 填,`updated_at` 由默认值刷新)。
- best-effort:写失败不打断阅读,记日志,下次前台或下次滚动停止再试。

### 4.2 读取(继续学习)
- "继续学习"入口(首页空间卡 / 路径页顶部)点击时,读该 `space_id` 在 `reading_progress` 的行 → 跳到 `unit_id` 并滚动到 `anchor`。
- 该行不存在(从没读过)→ 回退到原有默认行为(如该空间第一个未完成单元)。

## 5. 本地断点的迁移与取舍(避免新旧打架)

勘查清楚现有本地断点后:
- **服务端表成为唯一真源**。本地存储退化为离线兜底/缓存,或在迁移后弃用(按勘查结果定,报告你的方案)。
- **一次性迁移**:本次更新首次启动时,若本地存在断点而服务端该空间**无行**,把本地的推上去;若服务端**已有行**,用 `updated_at` 裁决——**服务端更新则以服务端为准,绝不能用陈旧的本地断点覆盖更新的网页端进度**。
- 迁移只跑一次,之后以服务端为准。
