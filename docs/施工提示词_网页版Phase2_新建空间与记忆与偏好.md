# 施工提示词 · 网页版 Phase 2(新建空间 + 关于我·记忆 + 阅读偏好)· 开工版

> 用途:粘贴给 **网页仓库**(Next.js)的 Claude Code。
> 栈:Next.js(App Router)+ `supabase-js` · 复用现有 Supabase(Postgres + RLS + Edge Functions)· 通义千问(经 `callLLM()`)。
> 前置:网页版 Phase 1 已上线(登录、首页、路径页、单元阅读、划词问 AI、提问记录、断点续学)。跨端续学已闭环。

> 本步范围:上述三块打包做全。**不做**路径手动编辑(拖拽/增删单元)、重新访谈、双端实时推送、付费/额度购买 UI。

## 0. 给 Claude Code 的前置说明

- **后端零改动原则**:三个访谈相关 Edge Function(`interview_next` / `summarize_understanding` / `gen_path`)与所有 prompt **原样复用,严禁修改**——App 依赖它们。网页端用用户会话 JWT 原样调用。若发现某函数契约与预期不符,**停下报告,不要改函数**。
- **唯一可能的后端增强**:阅读偏好的存储(见 §5 决策)。除此之外不动表结构、不动 RLS、不动 Edge Function。
- **先勘查、后动手**:访谈三函数的真实 I/O、`app_users.memory` 的真实结构,均需先在代码/DB 里查清并报告(见 §7 切片 0),别凭原型假设字段名。
- **密钥纪律**:客户端只用 Supabase URL + anon key;service-role key 不得出现在客户端。
- **增量交付,每片停下来让我验证**(见 §7)。
- 视觉与交互照搬原型;设计 token 沿用 Phase 1。

## 2. 新建空间:复用 App 的访谈闭环(照 App Step 3 的数据流)

流程(与 App 完全一致,只换 UI 到桌面):
```
新建空间(单一自适应输入)
   └─ 提交 → 访谈追问(loop: interview_next,history 放内存)──done──┐
                                                                    ▼
                                                summarize_understanding
                                                                    ▼
                                          理解确认页(可编辑卡片 + 我的推断)
                                                                    └─ 确认 ─┐
                                                                             ▼
                    落库:spaces + space_profiles(static) → gen_path(生成中)
                                                                             ▼
                              落库:phases + units(status='not_generated') → 进路径页(Phase 1 已有)
```

### 2.1 三个 Edge Function(原样调用,字段以勘查为准)
- `interview_next`:入参约 `{ task, initialInput, history:[{role,text}] }` → 出参含"继续追问 / 已问够"信号 + 下一问文案。**自适应**:用户说得多就快 done、说得少就多问。
- `summarize_understanding`:入参 `{ task, initialInput, history }` → 出参 `{ space_name, static:{goal/baseline/time/preferences…}, inferences:{learning_type/pace/depth…} }`。
- `gen_path`:入参 `{ task, spaceId }`(读该 space 的 `space_profiles.static` + learning_type)→ 出参 `{ phases:[{idx,title,units:[{idx,title}]}] }`。只生成阶段/单元标题,正文进入单元时由 Phase 1 现生成。
> ⚠️ 上述字段名是记忆中的形态,**以切片 0 勘查到的真实契约为准**。类型不限——后端四类模板已补齐,网页端**不做学习类型限制**。

### 2.2 UI(照原型)
- **新建空间**:单个大输入框 + 引导语("一句话或一段话都行……说得越具体,追问越少"),入口在首页"新建空间"卡(Phase 1 里标了 Phase 2 的那个)。
- **访谈对话**:桌面聊天列,一问一答,AI 打字态;history 放**内存,不落库**(与 App 一致)。物理键盘:`Enter` 发送、`Shift+Enter` 换行。
- **理解确认页**:`static` 渲染成**可编辑卡片**(目标/起点/时间/偏好,点"改"就地编辑);`inferences` 单独渲染成 **"我的推断,请纠正我"** 分区(学习类型/节奏/深度,各附一句推断理由 + 改判入口)。视觉上把"事实"与"推断"分开是建立信任的关键。

### 2.3 落库(照 App Step 3 §3)
| 时机 | 写入 |
|---|---|
| 确认理解 | `spaces` 一行(name=space_name,learning_type=inferences.learning_type)、`space_profiles` 一行(space_id,learning_type,static=确认后的 static,dynamic={}) |
| gen_path 返回 | `phases` 若干(idx/title,首阶段 active 其余 locked)、`units` 若干(idx/title/phase_id/space_id,status='not_generated',content=null) |
| 访谈对话 | 不落库 |

### 2.4 空间额度(复用既有,不新建逻辑)
- 新建空间前/落库时**复用既有的 `space_quota` 上限校验**(后端/RLS 若已约束,网页端只需捕获并**友好提示**"空间数量已达上限")。
- **不做**额度购买/邀请码兑换 UI(属商业化,后续);网页端只负责在超限时给体面的提示。

## 3. 关于我 · 跨空间记忆(编辑 user_memory)

「关于我」编辑的是**跨空间共享的用户记忆**(划词问 AI 时被注入的那份,记忆引擎会在后台自动补充)。**它不是某个空间的档案(space_profiles),不要混淆。**

- **先勘查**:确认这份记忆的真实存储(记忆中的形态是 `app_users.memory` 之类的 jsonb)、App 端如何读写、条目是否带分类(背景/职业/偏好/备注)。**按真实结构渲染,别硬套原型分组**——若底层是扁平数组则按 category 字段分组,若无分类则先平铺展示。报告结构与你的渲染方案后再动手。
- **操作**(照原型):分组列表;每条 hover 出现**编辑(就地改文本)/ 删除**;每组底部"添加一条"。
- **写入**:直接写回同一份记忆存储,`supabase-js` + RLS(自有行)。**改动即时同步手机端**(同一后端)。
- 与后台自动更新的关系:用户手动编辑直接生效,last-write-wins;不为此做额外合并逻辑(V1 从简)。
- 顶部文案明确:此记忆**跨所有空间共享、改动即时生效并同步手机**(照原型)。

## 4. 阅读偏好 + 桌面快捷键

### 4.1 偏好项(照原型,实时应用到单元阅读页)
- **正文字号**:小 / 中 / 大(约 15 / 16.5 / 18.5px)
- **栏宽**:窄 / 标准 / 宽(约 580 / 680 / 820px)
- **行距**:紧凑 / 标准 / 宽松(约 1.75 / 1.95 / 2.2)
- 改动实时作用于阅读页正文(原型右侧"实时预览"演示的即是效果)。

### 4.2 桌面快捷键(补齐到单元阅读页)
| 键 | 动作 |
|---|---|
| `⌘/Ctrl + K` | 选中文字后唤起问 AI(Phase 1 已实现,确认仍在) |
| `J` / `K` | 下一 / 上一单元 |
| `[` | 收起 / 展开左侧路径轨道 |
| `Esc` | 关闭问 AI 抽屉 |

### 4.3 偏好存储(§5 决策后实现)
见 §5:localStorage(每浏览器)或服务端 jsonb(随账号跨设备同步),二选一。

## 5. 决策项:阅读偏好存哪(需创始人拍板)

- **方案 A(推荐,on-thesis):服务端 additive jsonb 列** —— 在 `app_users` 加一个可空列 `web_prefs jsonb`(只新增、不改现有列、不影响 App),存 `{fontSize, width, lineHeight}`。优点:随账号**跨设备同步**(与产品"双端同源"一致),换台电脑偏好还在。
  ```sql
  alter table public.app_users add column if not exists web_prefs jsonb;
  ```
  (RLS 已在 `app_users` 上按自有行约束,无需新策略。)
- **方案 B(更省,零后端):localStorage** —— 只存在当前浏览器,不跨设备。适合想彻底零后端改动、且不在意偏好同步的情况。

> 推荐 A(一列可空 jsonb,是本 Phase 唯一后端触碰,极小且不影响 App)。请拍板,我据此实现 §4.3。

## 7. 建议开发顺序(每片停下来让我验证)

0. **勘查 + 报告(先做,停下确认)**:①三函数真实 I/O 字段;②`user_memory` 真实结构与 App 读写方式;③确认 §5 偏好存储方案。产出简短报告 + 各自实现方案,**等我确认再继续**。
1. **新建空间 + 访谈对话**:单输入 → 接 `interview_next`(history 内存)→ 桌面聊天列跑通。验证:说得详细少追问、说得少多追问。**不改 prompt,只验证 web 调用round-trip。**
2. **理解确认页**:接 `summarize_understanding` → 可编辑卡片 + "我的推断"分区。
3. **落库 + 生成路径**:确认 → 写 `spaces`+`space_profiles` → `gen_path` → 写 `phases`+`units` → 生成中过渡 → 跳 Phase 1 路径页。含额度超限友好提示。**端到端走一遍**:一句话 → 一直到路径页能点进新空间的单元现生成 + 划词问 AI。
4. **关于我·记忆**:按勘查结构渲染分组 → 就地编辑/删除/新增 → 写回记忆存储 → 确认手机端能看到改动。
5. **阅读偏好 + 快捷键**:三档实时作用于阅读页 + 按 §5 存储;补齐 `J/K`、`[`、`Esc`(`⌘K` 复核)。
6. **收尾**:访谈中断的最简恢复(重进即重来即可,与 App 一致)、错误/空/加载态。

## 8. 验收标准

- [ ] 从一句话(如"想学摄影,能拍好旅行风景人像,有微单但一直自动挡")出发,走完:追问 → 确认 → 生成路径 → 落到路径页。
- [ ] 说得详细时少追问、说得少时多追问(自适应,非固定问卷)。
- [ ] 确认页四张卡片可就地编辑;"我的推断"分区与事实卡片视觉分离,可改判学习类型。
- [ ] **生成的路径明显贴合该目标**(为"摄影入门·技能练习型"排,不是通用大纲);点进新空间任一单元能触发现生成 + 划词问 AI。
- [ ] 新建空间受 `space_quota` 约束,超限有友好提示(不报错崩溃)。
- [ ] 「关于我」能查看/编辑/删除/新增记忆条目;改动在手机端可见(同一后端)。
- [ ] 阅读偏好三档实时作用于正文;按 §5 方案保存(选 A 则换设备仍在)。
- [ ] 快捷键 `⌘K`/`J`/`K`/`[`/`Esc` 均可用。
- [ ] 全程无 service-role key;调用带用户 JWT;换账号看不到彼此的空间与记忆(RLS)。
- [ ] 未修改任何 Edge Function / prompt / 其他表(§5 选 A 的那一列除外)。

## 9. 明确不做(范围纪律)

- ✗ 路径手动编辑(拖拽排序 / 增删单元) —— 后续单排。
- ✗ 重新访谈 / 重置空间。
- ✗ 双端实时推送(Realtime,V3)。
- ✗ 付费体系 / 额度购买 / 邀请码兑换 UI(只复用既有额度上限校验)。
- ✗ 访谈对话持久化(内存即可,与 App 一致)。
- ✗ 修改 `interview_next`/`summarize_understanding`/`gen_path` 或其 prompt(契约不符只报告,不改)。
- ✗ 其他表结构 / RLS 改动(§5 选 A 的 `web_prefs` 列除外)。

如对实现有更优建议,先说明再改,不要默默扩大范围。
