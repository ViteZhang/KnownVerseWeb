# 商业化 Phase 3 · 策划(网页版付费 · Paddle)

> 性质:网页优先的商业化落地策划。付费主入口在网页(绕开 App 商店 15–30% 抽成),App 保留 IAP 为次要通道。
> 收款方案:已定 —— Merchant of Record「Paddle」。
> 定价:**已锁定**(见 §五);基于自建中转的单用户月成本 ≈ $0.5(平均活跃)测算。
> 上游:混合商业模型(月订阅为主 + 空间额度加购包),复用既有 `space_quota` 架构。

---

## 一、定位与原则

- **网页 = 主付费入口**。Stripe 因北京主体无法直接开户被排除;改用 Paddle(MoR),它不限制大陆主体卖家、代管全球税务、并同时覆盖 diaspora(卡/PayPal/Apple·Google Pay)与大陆用户(支付宝/微信)。
- **混合模型**:①月订阅(主收入轴)②空间额度加购包(增量 SKU,直接复用 `space_quota`)。App 端 IAP 保留,次要。
- **后端尽量复用**:权益驱动优先接到既有的 `space_quota.member_space_quota` 分量;计量复用既有 token 统计,翻译成用户可读单位。
- **纪律不变**:先验证"付费→权益"闭环这个灵魂链路,再铺付费墙 UI 与定价页。

---

## 二、收款架构(Paddle · 已定)

**为什么是 Paddle**:MoR 模式下,Paddle 以自己名义把软件转卖给终端用户,代收代缴全球销售税/VAT/GST、处理拒付退款、开发票,净额打款给你——绕开北京主体开不了 Stripe、以及跨境自行报税的两座大山。

**覆盖**:diaspora 用卡/PayPal/Apple·Google Pay;大陆用户用支付宝/微信。

**两个实操约束(写进执行)**:
1. **支付宝限制**:仅对"中国地址 + 人民币"的客户显示;**订阅续费单笔上限 1600 元人民币**(约 $220)。→ 本策划所有档位(年付 $69.99 ≈ ¥500 < 1600)均在上限内,无影响。
2. **打款与结汇**:Paddle 可全球打款(除受制裁国)。落到大陆主体涉及外汇结算;人在新加坡,可评估让 Paddle 打到 Wise/Payoneer/新加坡账户。**具体合规入账找会计确认**(非本策划的专业范围)。

**审批风险(提前备料)**:Paddle 的 KYB 审核可能慢、且有未公开门槛(有卖家被以"无历史交易记录"拒过)。→ 上线前把**退款政策、订阅条款、产品说明、公司资料**备齐提高一次过审率;**别把上线时间卡死在 Paddle 审批上**,留缓冲。

---

## 三、权益模型(entitlement · 要建的核心)

单一真源 + Paddle 事件驱动:

1. **`subscriptions` 表**(新增):user_id、status(active/past_due/canceled)、tier、current_period_end、paddle_subscription_id、updated_at。RLS 自有行。
2. **Paddle webhook → Supabase Edge Function → 写 `subscriptions`**:订阅创建/续费/取消/支付失败等事件到达 → 校验签名 → 更新该用户订阅态。这是"付费→权益"的灵魂链路,**第一个要验证的切片**。
3. **驱动既有额度**:订阅态映射到 `space_quota.member_space_quota`(会员=+N 空间),复用你早留好的这一分量;加购包映射到额度分量(按你现有 generated column 结构对齐)。
4. **前端读权益**:页面据 `subscriptions` + 计量(见四)决定是否放行/提示升级。

> 客户端只读权益结果,不碰 Paddle 密钥;webhook 校验与写库都在 Edge Function 完成。

---

## 四、计量与限额(免费档要能"数")

免费/付费的边界要落在**成本大头**上。经验判断:**单元生成**是重任务(成本高),划词问 AI 较轻,空间数已有 `space_quota` 天然可控。

- **计量对象**:每用户每月的 ①单元生成次数 ②提问次数 ③空间数(已有额度)。复用既有 token 统计后,翻译成用户可读单位(如"本月还剩 X 次深度提问")。
- **免费档目标成本**:每个免费用户月 LLM 成本压到 **≈ $0.15/月**以内(平均活跃用户是 $0.5,免费档要明显低于它)。据此设 caps:
  - **空间数**:1–2 个(体验到"建空间 + 被个性化追问"这个灵魂,但会自然想要更多)。
  - **单元生成 / 提问**:给月度上限(具体数字待"单次单元生成、单次提问"的分摊成本确定后卡死,但目标是把月成本压在 $0.15 内)。
- **换算逻辑**:一个年付用户净利 ≈ $48–60/年,可补贴约 **300 个**免费用户($0.15×12=$1.8/年)——免费池即便低转化也养得起。
- **闸口位置**:建空间(额度)、单元生成(次数)、提问(次数)三处,按成本权重设主闸;**单元生成为主闸**。

## 五、定价与打包(已锁定)

**成本假设**:自建中转,平均活跃用户 ≈ $0.5/月;付费用户为重度用户,按 **$1.5/月**保守测算。
**关键约束**:Paddle 抽 **5% + 每笔 $0.50**——固定 $0.50 在低价月付上抽成畸高(月付 $4.99→15%,月付 $8.99→10.6%,年付 $69.99→仅 5.7%)。→ 推论:**月付不低于 $5;主推年付**(一年一次交易费,抽成砍半,兼顾留存与现金流)。

### 定价表(USD)

| 方案 | 定价 | 折合月 | Paddle 抽成 | 净利/单位* |
|---|---|---|---|---|
| **Pro 年付**(主推·页面默认高亮) | **$69.99/年** | ~$5.83 | 5.7% | **~$48/年**(乐观 ~$60) |
| Pro 月付 | **$8.99/月** | — | 10.6% | ~$6.5/月(乐观 ~$7.5) |
| 空间加购包(一次性) | **+5 空间 $6.99**(≥$4.99 起) | — | ~11% | 高(未用前近零成本) |
| 免费档 | $0 | — | — | 每用户成本 ≤ ~$0.15/月 |

*净利 = 定价 − Paddle 费 − LLM 成本(付费按 $1.5/月保守;"乐观"按 $0.5/月)。

### 设计要点

- **年付 $69.99 是英雄款**:比月付 $8.99×12=$107.88 省约 **35%**(≈送 4 个月),把用户往年付推;对你抽成 5.7%、每用户年净利 $48–60、现金流一次到位、留存更好。页面默认高亮它。
- **月付 $8.99 的作用**是让年付显得更划算,同时保证真月付用户也有 10% 出头的健康抽成。**不得低于 $5**。
- **空间加购包 ≥ $4.99 起、打包卖**(如 +5 空间 $6.99)。$0.50 固定费让微交易($0.99/$1.99)抽成飙到 30%+,不划算;加购空间在被使用前几乎零边际成本,毛利极高。服务"不想订阅、只想多几个空间"的人。
- **单档 Pro 起步**:先不做多档分层(Pro/Max),验证转化后再分。
- **货币**:USD 打底(diaspora 通用),Paddle 自动本地化展示。

### 定位参照(消费级 AI 学习产品,网页价)

Duolingo Super 年付约合 $84–96/年(~$7/月);带 AI 对话的 Max 约 $168/年(~$14/月)。知识宇宙是 AI 原生、强个性化,价值定位接近 Max 那一档;年付 $69.99 处在"低于 Max、高于 Super"的合理带,对 diaspora 舒适。

### 两个后续杠杆(先留,不急做)

- **区域定价**:$69.99/年 ≈ ¥500,对 diaspora 舒适、对大陆偏高。Paddle 支持本地化定价(Duolingo 即美国 $96/年、部分地区约 $14–39/年)。大陆用户量起来后可单独设更低价,不影响 diaspora 主价。
- **年度促销**:上线一段后可用限时年付折扣(参考行业新年 50–60% off)冲一波转化。

## 六、付费墙与转化

- **转化时机**:用户建完 1–2 个空间、体验过被个性化追问(灵魂交互)后撞到额度墙——此刻升级意愿最高。付费墙文案对准"你已经体验到的价值"而非功能罗列。
- **升级点 UI**:额度用尽提示、定价页、订阅管理入口。**订阅管理复用 Paddle 托管的 customer portal**(退订/换卡/发票),省自建。
- **定价页 = SEO 资产**:用 Next.js 做公开定价页(当初选 Next.js 就是为这类公开页留的 SEO 余地),既是转化落点,也和"公开分享页获客"这条增长线共用一套公开页基建。

---

## 七、法务补全(卖订阅的最低面)

- Paddle 已替你担**全球税务 + 发票 + 大部分合规**。
- 你要补:**退款政策**、**订阅条款(自动续费的清晰披露 + 一键取消入口)**——多地消费者法硬要求,也是 Paddle 过审要看的。基于已有的隐私政策/用户协议增补即可。
- **结汇/入账**:北京主体收海外订阅收入的外汇处理,找会计确认(见二)。

## 八、分期建设(每片停下来验证)

0. **备料 + 注册 Paddle**(创始人):备齐退款/订阅条款/产品说明/公司资料,提交 Paddle KYB,开始审批(与开发并行,别阻塞)。
1. **灵魂链路:付费→权益闭环**:`subscriptions` 表 + Paddle webhook → Edge Function → 更新订阅态 → 驱动 `member_space_quota`。用 Paddle sandbox 跑通一笔订阅→权益生效→取消→权益回收。**先验证这个,再谈 UI。**
2. **计量与限额**:数单元生成/提问/空间;免费 caps 生效 + 撞墙拦截(目标 ≤$0.15/免费用户/月)。
3. **付费墙 UI**:额度用尽提示、升级引导、Paddle Checkout 接入、Paddle customer portal 接入。
4. **定价页(公开 + SEO)**:Next.js 公开定价页,展示 Pro 年付(高亮)/月付/空间包。
5. **收尾**:支付失败/续费失败的 dunning 提示、发票入口、边界态。

## 九、明确不做

- ✗ 不用 Stripe(主体不支持);不自建税务合规(Paddle 担)。
- ✗ V1 不做多档订阅分层(先单档 Pro 验证转化)。
- ✗ 月付不低于 $5;加购包不低于 $4.99(避开 Paddle 固定费坑)。
- ✗ 不在客户端放 Paddle 密钥;webhook 校验只在 Edge Function。
- ✗ 不改既有内容/学习功能;本 Phase 只加"付费—权益—计量—墙"。
- ✗ App 端 IAP 改造不在本 Phase(网页先跑通,App 次要通道后排)。
- ✗ V1 不做区域定价与促销(留作后续杠杆)。
